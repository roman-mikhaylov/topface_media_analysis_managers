# Анализ пролонгации договоров аккаунт‑менеджеров


## Описание задачи

Руководитель отдела сопровождения клиентов хочет оценить, насколько хорошо аккаунт‑менеджеры справляются с задачей пролонгации договоров с клиентами.  
Нужно рассчитать коэффициенты пролонгации по каждому менеджеру и по отделу в целом за 2023 год и собрать управленческий отчёт (таблица + дашборды). 

В компании используются два коэффициента:

- **K1** — пролонгация в первый месяц после окончания договора.  
- **K2** — пролонгация во второй месяц после окончания, при условии отсутствия пролонгации в первый месяц.

Расчёты ведутся:

- по отделу в целом;  
- по каждому аккаунт‑менеджеру (AM). [

## Исходные данные

Используются две CSV‑таблицы. 

### `financial_raw.csv`

Помесячные финансовые показатели по договорам.

- Строка — договор `id`.  
- Колонки: `Account`, `Причина дубля`, помесячные суммы и статусы (Ноябрь 2022 … Февраль 2024).  
- В ячейках — числовые суммы, а также спец‑значения: `стоп`, `end`, `в ноль`. 

### `prolongations_raw.csv`

Информация о договорах, подлежащих пролонгации.

- Строка — договор `id`.  
- Колонки: последний месяц реализации договора `month` и закреплённый аккаунт‑менеджер `AM`. 

Все расчёты выполняются на нормализованных версиях этих таблиц (`financial_long_clean.csv`, `prolongations_norm.csv`), которые формирует скрипт. 

## Бизнес‑логика расчётов (K1/K2)

Кратко, как считаются коэффициенты (подробно — в `code_and_logic.md`). 

- Базовым считается месяц окончания договора `last_month_code` из `prolongations_raw.csv`.  
- Договоры, в которых в любом месяце встречается `стоп` или `end`, полностью исключаются из пролонгаций.  
- Для K1 используются договоры, заканчивающиеся в месяце \(L = M-1\); для K2 — в месяце \(L = M-2\), причём для K2 исключаются договоры с отгрузкой в первый месяц после окончания. 

Коэффициенты на уровне отдела:

- \(K1(M) = \text{факт пролонгации в M} / \text{база договоров с L = M-1}\).  
- \(K2(M) = \text{факт пролонгации в M} / \text{база договоров с L = M-2, не пролонгированных в M-1}\). 

На уровне AM формулы те же, но в разрезе конкретного менеджера. 

## Структура репозитория

```text
topface_media_analysis_managers/
  data/
    financial_raw.csv
    prolongations_raw.csv
    ... (нормализованные *.csv создаются автоматически)
  out/
    department_from_prol_2023.csv
    am_from_prol_2023.csv
    am_from_prol_k1_pivot_2023.csv
    am_from_prol_k2_pivot_2023.csv
  py/
    manager_analysis.py
  code_and_logic.md
  README.md
```


- `code_and_logic.md` — подробное описание логики, допущений и полный код с комментариями.  
- `py/manager_analysis.py` — исполняемый скрипт, который автоматизирует все шаги: нормализацию данных, фильтрацию, расчёт K1/K2 и выгрузку отчётов.

## How to run

### 1. Клонировать репозиторий

```bash
git clone https://github.com/roman-mikhaylov/topface_media_analysis_managers.git
cd topface_media_analysis_managers
```


### 2. Создать и активировать виртуальное окружение

macOS / Linux:

```bash
python3 -m venv .venv
source .venv/bin/activate
```

Windows (PowerShell):

```powershell
python -m venv .venv
.\.venv\Scripts\activate
```


В редакторе (например, VS Code) выберите интерпретатор из `.venv` перед запуском скрипта. [support.microsoft](https://support.microsoft.com/en-us/office/save-your-workbook-92e4aae0-452d-497f-a470-570610ff720a)

### 3. Установить зависимости

Проект использует библиотеку **pandas** для обработки табличных данных.

```bash
pip install --upgrade pip
pip install pandas
```


Для экспорта в Excel/работы с `.xlsx` при необходимости:

```bash
pip install openpyxl
```


### 4. Подготовить входные файлы

В папке `data/` должны находиться:

- `financial_raw.csv` — исходная таблица финансовых данных;  
- `prolongations_raw.csv` — таблица договоров с последним месяцем реализации и ответственным AM. 

Структура:

```text
topface_media_analysis_managers/
  data/
    financial_raw.csv
    prolongations_raw.csv
```


### 5. Запустить расчёты

Из корня репозитория, в активном виртуальном окружении:

```bash
python py/manager_analysis.py
```


Скрипт:

- нормализует `financial_raw.csv` и `prolongations_raw.csv`;  
- исключает договоры со статусом `стоп / end`;  
- рассчитывает базы и факты пролонгаций (K1/K2) по отделу и по каждому AM;  
- сохраняет агрегированные таблицы в папку `out/`. 

### 6. Результат выполнения

После успешного запуска в `out/` появятся файлы:

- `department_from_prol_2023.csv` — показатели баз и коэффициентов K1/K2 по отделу по месяцам;  
- `am_from_prol_2023.csv` — те же показатели в разрезе AM × месяц;  
- `am_from_prol_k1_pivot_2023.csv` — матрица K1 по AM (строки — менеджеры, столбцы — месяцы);  
- `am_from_prol_k2_pivot_2023.csv` — аналогичная матрица по K2.

Эти файлы используются как источники данных для итогового отчёта и дашбордов в Google Sheets / Excel. 

## Отчёт для руководителя

Итоговый управленческий отчёт собирается в Google Sheets / Excel на основе файлов из `out/`:

- лист «Весь отдел» — динамика K1/K2 по месяцам;  
- лист «Менеджеры» — показатели по каждому AM;  
- дашборды: диаграммы с динамикой коэффициентов, сравнение менеджеров, годовые итоги. 

Готовый файл отчёта (`.xlsx`) можно положить, например, в папку `report/` и сослаться на него из README:

```markdown
- `report/topface_managers_retention_report_2023.xlsx` — управленческий отчёт с дашбордами.
```
